# Infrastructure Spec

## Overview

Deploy the monorepo apps (web + worker) on two Hetzner Cloud VMs running k3s.
Target: minimal cost, simple operations, good enough for production-light workloads.

## Nodes

| Role | Hetzner Type | Specs | Monthly Cost |
|------|-------------|-------|-------------|
| server | CX22 | 2 vCPU, 4 GB RAM, 40 GB disk | ~€4.5 |
| agent | CX22 | 2 vCPU, 4 GB RAM, 40 GB disk | ~€4.5 |

Total compute: ~€9/mo.

**Node 1 (server):** k3s control plane + Traefik ingress + web app + SQLite. 

**Node 2 (agent):** k3s agent + worker app.

> ARM (CAX11, ~€3.99/mo) is cheaper but requires multi-arch Docker builds.
> Start with x86 (CX22) to keep builds simple. Revisit ARM later.

## Database

SQLite, stored on a persistent volume on the server node.

No managed DB, no Postgres container. SQLite is sufficient for the tutorial workload
and removes an entire service from the cluster.

### Backups

Cron job (k8s CronJob or host-level cron) copies the SQLite file to
Hetzner Object Storage (S3-compatible) on a schedule (e.g. every 6 hours).

- Hetzner Object Storage: ~€0.006/GB/mo
- A few MB of SQLite snapshots = essentially free

Backup script outline:
```bash
sqlite3 /data/app.db ".backup /tmp/backup.db"
s3cmd put /tmp/backup.db s3://monorepo-backups/app-$(date +%Y%m%d-%H%M%S).db
```

## Container Registry

GitHub Container Registry (ghcr.io) — free for public repositories.

Images are built in GitHub Actions and pushed to ghcr.io.

## Networking

- Traefik (ships with k3s) as ingress controller
- No domain initially — access via node public IP
- Hetzner firewall:
  - 80, 443: open to all (public traffic)
  - 6443: open to all (k3s API — TLS + client certificate auth, same model as managed k8s)
  - 22: restricted to your IP only
- Inter-node communication over Hetzner private network (free, same DC)

## Deployment (CD)

GitHub Actions runs `kubectl apply` over SSH tunnel or via kubeconfig.
Zero RAM/CPU overhead on the cluster — no in-cluster CD tooling.

```
push to main → build images → push to ghcr.io → kubectl apply -f k8s/
```

The workflow:
1. CI builds and pushes tagged images to ghcr.io
2. CD job opens SSH tunnel to server node
3. Runs `kubectl set image` or `kubectl apply` with new image tag
4. k3s pulls the image and rolls out

> **Future option:** If GitOps becomes desirable, Flux v2 fits in ~128-256 MB RAM
> and watches the repo for manifest changes. Not needed initially.

## Provisioning (Terraform)

Terraform with the Hetzner Cloud provider (`hetznercloud/hcloud`).

### Resources (~5 total, ~100 lines HCL)

| Resource | Purpose |
| -------- | ------- |
| `hcloud_ssh_key` | Your public key for SSH access |
| `hcloud_network` + `hcloud_network_subnet` | Private network between nodes (free, same DC) |
| `hcloud_firewall` | Allow 22, 80, 443, 6443 |
| `hcloud_server` (server) | Node 1 — k3s server, cloud-init installs k3s |
| `hcloud_server` (agent) | Node 2 — k3s agent, cloud-init joins cluster |

### Sequencing

Node 2 needs a join token generated by Node 1 during k3s install.

1. Terraform creates Node 1 with cloud-init that installs k3s server
2. `remote-exec` provisioner waits for k3s to be ready, reads the join token from `/var/lib/rancher/k3s/server/node-token`
3. Token is passed to Node 2's cloud-init via `depends_on` on Node 1
4. Node 2 installs k3s agent and joins the cluster

### Terraform State

Stored in Hetzner Object Storage (S3-compatible) — same bucket region as backups.
Never committed to git. Encrypted at rest by Hetzner.

```hcl
# backend.tf
terraform {
  backend "s3" {
    bucket   = "monorepo-tfstate"
    key      = "infra/terraform.tfstate"
    endpoint = "https://fsn1.your-objectstorage.com"
    region   = "fsn1"
  }
}
```

### Post-apply steps

1. Copy kubeconfig from Node 1 (Terraform can output this)
2. Store kubeconfig as GitHub Actions secret for CD (scoped ServiceAccount, not cluster-admin)

```
infra/
├── main.tf
├── variables.tf
├── outputs.tf          # kubeconfig, server IP, agent IP
└── cloud-init/
    ├── server.yaml     # installs k3s server
    └── agent.yaml      # installs k3s agent (templated with token)
```

## Security

- **Branch protection on main** — require reviews, no direct push. The repo is the deployment authority; protect it accordingly.
- **Firewall** — SSH restricted to your IP. k8s API (6443) open but secured by TLS client certificates (same as managed k8s). 80/443 open for traffic.
- **CD credentials** — scoped ServiceAccount in k3s (deploy-only, not cluster-admin). Stored as GitHub Actions secret.
- **Terraform state** — remote backend in Hetzner Object Storage, never in git.
- **Backup credentials** — S3 keys stored as k8s Secrets, referenced by the CronJob. Stay in-cluster.
- **No TLS yet** — known gap. Traefik + Let's Encrypt is the natural next step once a domain is added.
- **Public images** — repo is public, so ghcr.io images are public too. No pull secrets needed in the cluster.
